# Техническая документация системы "Намерения"

Этот документ описывает текущую реализацию системы Intentions в коде Space Station 14: какие файлы за что отвечают, как данные проходят от YAML-шаблонов до интерфейса игрока, где находится серверная логика, как устроены проверка контента, снимок раунда, условия, волны распределения, запись в runtime, жизненный цикл, refill, reveal, интерфейс и отладочные команды.

Документ написан как технический справочник для C# разработчика, который хочет поддерживать или расширять систему. Он описывает текущее состояние кода, а не будущий идеальный дизайн.

Рядом с ним лежит более простой обзорный документ для чтения без погружения в код: `Документация системы Намерения.md`.

## 1. Степень независимости от кода Space Station 14

Реализация системы "Намерения" в основном добавлена рядом с существующим кодом Space Station 14 и не переписывает базовые игровые механики. Большая часть нового кода находится в трех изолированных областях:

- `Content.Shared/Intentions`;
- `Content.Server/Intentions`;
- `Content.Client/Intentions`.

Это означает, что система сделана как новая подсистема поверх существующей игры. Она использует штатные механизмы Space Station 14, но не меняет работу профессий, antagonist objectives, round rules, выбор персонажа, сохранение между раундами или базовую базу данных. Runtime-состояние "Намерений" хранится только в памяти текущего раунда, поэтому для него не нужны миграции базы данных и нет cross-round persistence.

Влияние на исходный код Space Station 14 можно оценить как низкое или умеренное. Низкое, потому что почти все новые правила распределения, проверки и интерфейсы лежат в отдельных папках `Intentions`. Умеренное, потому что для настоящей работы в живом раунде система все равно читает несколько существующих игровых систем: mind, профессии, игровые сессии, станцию, время раунда, состояние игроков, EUI и события очистки раунда.

Точки интеграции с базовой игрой:

- SS14 prototypes: шаблоны категорий, сценариев и намерений описаны как прототипы и загружаются через `IPrototypeManager`.
- Fluent-локализация: текст карточек, скрытых карточек, интерфейса и тестового контента хранится в `.ftl`.
- `EntitySystem`: серверные и клиентские владельцы логики оформлены как системы RobustToolbox.
- `MindComponent` и `SharedMindSystem`: система определяет, у какого игрока есть mind, тело, смерть, роли и цели.
- `GameTicker`: используется для режима игры, времени раунда, профиля игрока и статуса раунда.
- `StationSystem`: используется для имени станции.
- `SharedJobSystem`: используется для профессии и отдела кандидата.
- Player sessions: snapshot берет только игровых кандидатов с сессией в игре.
- EUI: пользовательский интерфейс карточек открывается через штатный EUI.
- Admin verbs: админский просмотр добавлен через verb для администратора.
- Console commands: отладка и ручной запуск волн доступны через консольные команды.
- `RoundRestartCleanupEvent`: runtime registry очищается при перезапуске раунда.
- `PlayerSpawnCompleteEvent`: стартовая автоматическая волна планируется после завершения спавна игрока; если момент неудобный, запуск подхватывается через `GameRunLevelChangedEvent` и резервную проверку в `IntentionsDistributionSystem.Update`.

Система пока не меняет:

- antagonist objectives;
- выбор профессии;
- правила запуска раунда;
- persistence между раундами;
- базовую механику latejoin;
- награды, действия и postconditions.

Основная мысль для поддержки: Intentions можно читать как отдельный модуль. При разборе багов обычно сначала нужно смотреть папки `Content.Shared/Intentions`, `Content.Server/Intentions`, `Content.Client/Intentions`, а в старый код SS14 идти только за точками интеграции.

## 2. Общая архитектура

Архитектура разделена по стандартной границе Space Station 14: shared, server и client.

`Content.Shared/Intentions` содержит общие типы данных. Здесь лежат prototype definitions, модели проверки, snapshot-модели, predicate engine, модели волн, runtime-модели и модели UI. Shared-слой не должен читать живое серверное состояние и не должен зависеть от client/server-only классов. Он нужен, чтобы сервер, клиент и тесты говорили на одном языке: одинаково понимали `ScenarioInstance`, `IntentionInstance`, `CandidateFacts`, `IntentionsEuiState` и другие структуры.

`Content.Server/Intentions` содержит авторитетную серверную логику. Сервер проверяет контент, строит снимок раунда, выбирает сценарии, собирает слоты, записывает runtime-состояние, следит за пропавшими владельцами, раскрывает hidden-намерения по таймеру, формирует read-model для UI и обслуживает отладочные команды. Вся логика, которая может изменить состояние раунда или выдать игроку карточку, находится на сервере.

`Content.Client/Intentions` содержит только клиентский интерфейс. Клиент показывает окно, получает готовое состояние через EUI, дает выбрать карточку и открыть материалы. Обычный player UI больше не опирается на отдельную кнопку ручного refresh: открытое окно обновляется сервером после commit и reveal. Клиент не выбирает сценарии, не проверяет условия и не решает, что скрыто или раскрыто. Это важно: если hidden-намерение скрыто, клиент получает уже очищенную карточку, а не полные данные с флагом "не показывать".

Основной поток данных:

1. YAML-шаблоны загружаются как SS14 prototypes.
2. `IntentionsValidationService` проверяет их и строит `ValidationCatalog`.
3. `IntentionsSnapshotService` строит неизменяемый snapshot текущего раунда.
4. `IntentionsWaveOrchestrator` считает квоты и выбирает сценарии.
5. `IntentionsScenarioBuilder` собирает слоты выбранного сценария.
6. `IntentionsCommitService` атомарно записывает результат в `IntentionsRuntimeRegistry`.
7. `IntentionsQueryService` строит read-model для UI.
8. Server EUI отправляет read-model клиенту.
9. Client EUI показывает карточки игроку или администратору.

Такое разделение защищает несколько важных инвариантов:

- builder не имеет side effects;
- UI не является источником истины;
- hidden-данные не раскрываются клиенту раньше времени;
- commit можно откатить при ошибке;
- refill может считать дефицит по runtime-состоянию, а не по UI.

## 3. Карта файлов

### `Content.Shared/Intentions/Prototypes`

Эта папка описывает типы YAML-прототипов.

- `IntentionTemplatePrototype.cs` описывает шаблон одного намерения: тип намерения, автора, ключи локализации, видимость по умолчанию, цвет, иконку, теги, дату создания.
- `ScenarioCategoryPrototype.cs` описывает категорию сценариев: квоты, приоритет, лимит основных намерений на mind, цвет и иконку категории.
- `ScenarioTemplatePrototype.cs` описывает сценарий: категорию, вес, включенность, глобальные условия и набор слотов `ScenarioEntry`.

В этих файлах также находятся вложенные data definitions, которые используются внутри YAML: `ScenarioEntry`, `QuotaRule`, `PredicateDefinition`, `CompareToDefinition`, `VisibilityOverrideDefinition`, `RevealDefinition`, `TextParameterBindingDefinition`.

### `Content.Shared/Intentions/Validation`

Эта папка содержит модели результата проверки.

- `ValidationCatalog` хранит только валидные категории, намерения и сценарии.
- `ValidatedScenarioTemplate` хранит проверенный сценарий вместе с заранее рассчитанным `SlotBuildOrder`.
- `ValidationIssue` описывает ошибку, предупреждение или информационное сообщение проверки.
- `ValidationIssueSeverity` и `ValidationObjectType` нужны для классификации результата.

Сам сервис проверки находится на сервере, но модели результата общие, потому что ими удобно пользоваться в тестах, debug tooling и read-only диагностике.

### `Content.Shared/Intentions/Snapshot`

Эта папка описывает снимок раунда.

- `IntentionsSnapshotRequest` задает тип снимка: start или refill.
- `IntentionsSnapshot` хранит immutable-состояние одной волны.
- `RoundFacts` хранит факты раунда: режим, время станции, имя станции, численность экипажа, число security и event tags.
- `CandidateFacts` хранит факты одного возможного участника.
- `AntagSummary` хранит агрегированную информацию об антагонистах.
- `SnapshotBuildResult` и `SnapshotBuildIssue` описывают успех или ошибки сборки снимка.
- `IntentionsSnapshotFactory` нормализует сырые данные, сортирует кандидатов и собирает неизменяемые коллекции.

### `Content.Shared/Intentions/Predicates`

Эта папка отвечает за условия.

- `IntentionsPredicateSchema` содержит единый список допустимых полей и операторов.
- `IntentionsPredicateEngine` чисто проверяет условия по snapshot и выбранным слотам.
- `PredicateEvaluationResult` возвращает успех, false-result или ошибку runtime-формы.
- `PredicateRejectReason` объясняет, почему условие не прошло.

### `Content.Shared/Intentions/Waves`

Эта папка содержит модели волн распределения.

- `IntentionsStartWaveRequest` описывает стартовую волну.
- `IntentionsRefillWaveRequest` описывает refill-волну.
- `StartWaveResult` возвращает итог волны.
- `DistributionWaveContext` хранит состояние волны: seed, snapshot id, kind, квоты, успешные сборки, reject reasons.
- `CategoryWaveState` хранит состояние одной категории внутри волны.
- `ScenarioBuildResult` и `ScenarioSlotBuildResult` описывают результат сборки сценария.
- `ScenarioRejectReason` и `SlotRejectReason` объясняют отказы.

### `Content.Shared/Intentions/Runtime`

Эта папка содержит runtime-модели, которые записываются в registry.

- `ScenarioInstanceUid` и `IntentionInstanceUid` являются round-scoped идентификаторами.
- `ScenarioInstance` описывает созданный в раунде сценарий.
- `ScenarioSlotAssignment` описывает назначение конкретного слота.
- `IntentionInstance` описывает конкретную карточку намерения у конкретного mind.
- `ScenarioRuntimeStatus`, `ScenarioSlotAssignmentStatus`, `IntentionRuntimeStatus` описывают состояния runtime-объектов.
- `IntentionRevealMode` описывает режим раскрытия hidden-намерения.
- `CommitScenarioBuildResult` описывает успех или ошибку commit.

### `Content.Shared/Intentions/UI`

Эта папка содержит сетевые модели EUI.

- `IntentionsEuiState` описывает весь экран намерений.
- `IntentionsCardView` описывает одну карточку, включая локализованные тексты, автора, статус, иконку, цвет и уже вычисленные параметры подстановки.
- `IntentionsScenarioAdminView` описывает read-only данные сценария для администратора.
- `IntentionsEuiMode` разделяет player view и admin view.
- EUI-сообщения используются для действий окна и открытия материалов. Ручной refresh в обычном player UI больше не используется.

### `Content.Server/Intentions/Validation`

- `IntentionsValidationService.cs` проверяет прототипы через `IPrototypeManager`, читает loc keys через localization resolver и возвращает `ValidationCatalog`.

### `Content.Server/Intentions/Snapshot`

- `IntentionsSnapshotService.cs` читает живое состояние игры и собирает сырые факты для snapshot.

### `Content.Server/Intentions/Waves`

- `IntentionsWaveOrchestrator.cs` управляет start/refill волнами.
- `IntentionsScenarioBuilder.cs` собирает слоты одного сценария.
- `IntentionsQuotaCalculator.cs` считает квоты и лимиты по категории.
- `IntentionsDeterministicRandom.cs` дает стабильный seed и deterministic weighted pick.
- `IntentionsDistributionSystem.cs` владеет автоматическим запуском стартовой волны, планировщиком refill-волн, безопасным ручным запуском, серверными логами, звуковым уведомлением и приватным чат-уведомлением о новых намерениях.

### `Content.Server/Intentions/Runtime`

- `IntentionsRuntimeRegistry.cs` хранит runtime-состояние раунда и индексы.
- `IntentionsRuntimeSystem.cs` владеет registry как серверная система и сбрасывает его при очистке раунда.
- `IntentionsRuntimeAdminService.cs` содержит runtime admin mutations: physical reset/remove, forced assign и принудительное reveal hidden intentions.
- `IntentionsCommitService.cs` атомарно записывает successful build result в registry.
- `IntentionsLifecycleModels.cs` содержит модели доступности владельца.
- `IntentionsLifecycleService.cs` реализует переходы `Missing`, `Frozen`, `Cancelled`.
- `IntentionsLifecycleSystem.cs` связывает lifecycle service с событиями mind/player и периодической проверкой.
- `IntentionsRevealService.cs` раскрывает hidden-намерения по таймеру.
- `IntentionsRevealSystem.cs` периодически вызывает reveal service и обновляет UI.

### `Content.Server/Intentions/UI`

- `IntentionsQueryService.cs` строит hidden-safe read-model.
- `IntentionsEui.cs` является серверной EUI-оберткой.
- `IntentionsUiSystem.cs` открывает player/admin UI и обновляет состояние.
- `IntentionsAdminVerbSystem.cs` добавляет read-only admin verb.
- `IntentionsAssignmentNotificationService.cs` агрегирует newly committed intentions по `mind`, строит hidden-safe текст сообщения и отправляет приватное server-chat уведомление только затронутому игроку.

### `Content.Server/Intentions/Debug`

- `IntentionsDebugCommands.cs` содержит консольные команды.
- `IntentionsDebugFormatters.cs` форматирует validation, wave, registry, reveal и trace-вывод.

### `Content.Client/Intentions/UI`

- `IntentionsUIController.cs` связывает окно намерений с кнопкой в верхнем меню игры.
- `IntentionsUiSystem.cs` хранит и переиспользует клиентский экземпляр player UI.
- `IntentionsEui.cs` показывает окно намерений, список карточек и окно материалов.
- `IntentionsListCardControl.xaml` и `.cs` описывают одну строку списка карточек.
- `IntentionsTextHighlighting.cs` выделяет уже подставленные значения в видимом тексте карточки.
- `GameTopMenuBar.xaml` и `GameTopMenuBarUIController.cs` добавляют отдельную кнопку открытия Intentions в верхнюю панель.

### `Content.Tests/.../Intentions`

Тесты покрывают validation, predicates, snapshot, waves, builder, commit, registry, lifecycle, refill, reveal, query, debug tooling и smoke content.

## 4. Шаблоны и прототипы

Шаблоны системы находятся в YAML и загружаются штатным prototype manager Space Station 14. В текущем коде используются три основных prototype type:

- `intentionTemplate`;
- `scenarioTemplate`;
- `scenarioCategory`.

`IntentionTemplatePrototype` описывает не выданную карточку, а шаблон будущей карточки. В нем находятся:

- `kind`: `primary` или `secondary`;
- `author`: простая строковая подпись автора карточки, которая показывается в UI;
- `nameLoc`: ключ названия;
- `summaryLoc`: короткое описание для списка карточек;
- `descriptionLoc`: основной текст;
- `oocInfoLoc`: внеигровое пояснение, которое показывается перед полным описанием;
- `copyableTextLoc`: текст, который можно скопировать;
- `hiddenLabelLoc`: подпись скрытой карточки;
- `defaultVisibility`: видимость по умолчанию;
- `icon`, `color`, `tags`, `creationDate`.

Для `nameLoc` и `summaryLoc` текущий валидатор проверяет ограничение в 35 символов на локализованную строку.

`icon` у `intentionTemplate` и `scenarioCategory` хранится как ссылка на спрайт через `sprite/state`, а не как произвольная строка.

`ScenarioCategoryPrototype` группирует сценарии. Категория задает, сколько сценариев такого типа надо выдать и как часто один mind может быть владельцем primary-намерения в этой категории. Основные поля:

- `quotaByGameMode`: квота по режиму игры, включая обязательный `default`;
- `maxPrimaryPerMindByGameMode`: лимит primary-намерений на mind, включая обязательный `default`;
- `priority`: порядок обработки категории;
- `icon` и `color`: визуальные значения для UI;
- `description`: техническое описание.

`ScenarioTemplatePrototype` описывает целый сценарий. Сценарий является главной единицей распределения. Система не раздает отдельные `intentionTemplate` напрямую: сначала выбирается сценарий, затем внутри него собираются слоты, затем для каждого собранного слота создается `IntentionInstance`.

Важные поля сценария:

- `category`: ссылка на `scenarioCategory`;
- `enabled`: выключенные сценарии валидируются, но не попадают в pool распределения;
- `weight`: вес при случайном выборе внутри категории;
- `globalPredicates`: условия на весь раунд;
- `entries`: слоты сценария.

`ScenarioEntry` описывает один слот внутри сценария. Слот указывает:

- `slotId`: технический идентификатор слота;
- `kind`: `primary` или `secondary`;
- `required`: обязателен ли слот;
- `intentionId`: какой `intentionTemplate` будет создан для этого слота;
- `candidatePredicates`: условия на кандидата;
- `bindToSlot`: взять того же кандидата, что и в другом слоте;
- `allowSameActorAs`: разрешить повторное использование кандидата из указанного слота;
- `visibilityOverride`: переопределить видимость;
- `textParameterBindings`: заранее вычислить параметры текста.

`QuotaRule` поддерживает MVP-режимы:

- `fixed`: фиксированное число;
- `ratio`: доля от базовой численности экипажа;
- `clamp`: доля от базовой численности экипажа, зажатая в диапазон `min..max`.

`PredicateDefinition` описывает одно условие. Оно содержит `scope`, `field`, `operator`, значения и, если нужно, `compareTo`.

`TextParameterBindingDefinition` связывает параметр локализованного текста с источником:

- `self`: текущий кандидат слота;
- `slot`: кандидат другого слота;
- `round`: факт раунда;
- `literal`: буквальное значение из YAML.

`RevealDefinition` в MVP поддерживает `none` и `timer`. `manual` и `condition` считаются будущими расширениями и не входят в текущий MVP.

Связь с FTL простая: YAML хранит ключи, а реальные строки лежат в Fluent-файлах. При commit параметры текста вычисляются один раз и затем используются для локализации карточки.

## 5. Валидация

`IntentionsValidationService` находится в `Content.Server/Intentions/Validation/IntentionsValidationService.cs`. Это первый обязательный шлюз перед runtime. Его задача - взять все prototype objects, проверить их и вернуть `ValidationCatalog`, в который попадут только валидные объекты.

Сервис проверяет:

- обязательные поля;
- ссылки между категориями, сценариями и намерениями;
- наличие FTL-ключей;
- ограничения длины локализованных строк;
- корректность `kind`;
- корректность цветов и дат;
- квоты категорий;
- структуру сценария;
- ровно один owner slot;
- соответствие `ScenarioEntry.kind` и `IntentionTemplate.kind`;
- допустимость `globalPredicates` и `candidatePredicates`;
- связи `bindToSlot`, `allowSameActorAs`, `compareTo`;
- text bindings;
- reveal и visibility;
- граф зависимостей слотов.

Из последних ограничений, которые особенно важно помнить при работе с контентом:

- `nameLoc` для `intentionTemplate` не должен быть длиннее 35 символов;
- `summaryLoc` для `intentionTemplate` тоже не должен быть длиннее 35 символов.

`ValidationCatalog` хранит:

- `ValidIntentions`;
- `ValidCategories`;
- `ValidScenarios`;
- `ValidCategoryOrder`;
- `ValidScenarioOrder`;
- `Issues`.

Порядок объявления важен для детерминизма. Если две категории имеют одинаковый `priority`, система использует declaration order из `ValidCategoryOrder`. Если сценарии выбираются с одинаковыми условиями, их исходный порядок также остается воспроизводимым.

`ValidatedScenarioTemplate` хранит исходный `ScenarioTemplatePrototype` и `SlotBuildOrder`. `SlotBuildOrder` вычисляется заранее во время validation, чтобы runtime builder не решал граф на лету. Это снижает риск runtime-ошибок и делает сборку сценария проще: builder идет по уже проверенному порядку.

Граф зависимостей строится из:

- `bindToSlot`;
- `allowSameActorAs`;
- `compareTo.slotId`.

Owner не должен зависеть от других слотов и должен быть первым. Циклы запрещены. При нескольких корректных порядках сохраняется порядок записи `entries`.

Если объект невалиден, он не попадает в `ValidationCatalog`. Ошибка при одном сценарии не должна ломать весь каталог, если остальные шаблоны валидны.

## 6. Snapshot

Snapshot нужен, чтобы волна распределения работала по одному неизменяемому состоянию раунда. Без snapshot builder мог бы читать живую игру на каждом слоте: один игрок сменил тело, другой умер, режим поменялся, и внутри одной волны получились бы разные ответы. Snapshot делает волну воспроизводимой.

`IntentionsSnapshotService` находится в `Content.Server/Intentions/Snapshot/IntentionsSnapshotService.cs`. Он является `EntitySystem` и читает живую игру:

- текущий режим через `GameTicker`;
- длительность раунда через `GameTicker.RoundDuration()`;
- имя станции через `StationSystem`;
- игровые сессии через player manager;
- mind и тело через mind-системы и компоненты;
- профессию и отдел через `SharedJobSystem`;
- профиль персонажа через `GameTicker.GetPlayerProfile(session)`;
- возраст, species и sex через `HumanoidProfileComponent`;
- traits из профиля игрока;
- mindshield через `MindShieldComponent`;
- antag roles через `MindRoleComponent`;
- ghost-role antag через `GhostRoleMarkerRoleComponent`;
- objective types через objectives mind.

`IntentionsSnapshotRequest` задает, зачем строится snapshot:

- `Start(waveId)` для стартовой волны;
- `Refill(waveId)` для refill-волны.

`IntentionsSnapshotFactory` отделяет сбор живых ECS-данных от нормализации. Сервис собирает сырые факты, а фабрика:

- стабильно сортирует кандидатов;
- считает `crewCount`;
- считает `securityCount`;
- собирает `AntagSummary`;
- копирует коллекции в immutable-структуры;
- строит быстрые индексы по mind.

Порядок кандидатов стабильный: сначала `UserId.ToString()` ordinal, затем `MindId`. Это важно для deterministic selection, потому что одинаковый seed и одинаковое состояние должны давать одинаковую волну.

`RoundFacts` содержит:

- `GameMode`;
- `StationTime`;
- `StationName`;
- `CrewCount`;
- `SecurityCount`;
- `EventTags`;
- `AntagSummary`.

`CandidateFacts` содержит:

- `MindId`;
- `OwnerEntityUid`;
- `UserId`;
- `CharacterName`;
- `Job`;
- `Department`;
- `Age`;
- `Species`;
- `Sex`;
- `Traits`;
- `HasMindshield`;
- `AntagRoles`;
- `AntagObjectiveTypes`.

`AntagSummary` в текущей реализации считает `ghostRoleAntagCount` не только по `GhostRoleMarkerRoleComponent`. В `IntentionsSnapshotService` есть локальный Intentions-only override для крупных ghost-role antag roles, чтобы снапшот не терял их при mixed-семантике ролей. Сейчас туда входят:

- `Dragon`;
- `SpaceNinja`;
- `ParadoxClone`;
- `Nukeops`;
- `NukeopsCommander`;
- `NukeopsMedic`;
- `Wizard`;
- `SubvertedSilicon`;
- `MothershipCore`;
- `Xenoborg`.

Это влияет только на snapshot/read-model уровень Intentions и не меняет базовую модель ролей SS14.

Если snapshot невозможно построить из-за невозможной структуры, результат возвращается как `SnapshotBuildResult` с ошибками. Волна должна трактовать это как блокирующую проблему, а не пытаться распределять намерения по неполному состоянию.

## 7. Predicate Engine

Predicate engine - это чистая проверка условий. Он не читает ECS, не меняет registry и не вызывает UI. Входом являются уже загруженные predicate definitions, snapshot и, для candidate predicates, уже выбранные слоты текущего сценария.

Основные файлы:

- `Content.Shared/Intentions/Predicates/IntentionsPredicateSchema.cs`;
- `Content.Shared/Intentions/Predicates/IntentionsPredicateEngine.cs`;
- `Content.Shared/Intentions/Predicates/PredicateEvaluationResult.cs`.

`IntentionsPredicateSchema` задает единый список допустимых полей и операторов. Validation и runtime используют одну схему, чтобы не было ситуации, когда validation разрешает поле, а runtime его не понимает.

Поддерживаемые операторы:

- `equals`;
- `notEquals`;
- `in`;
- `notIn`;
- `contains`;
- `notContains`;
- `>`;
- `>=`;
- `<`;
- `<=`;
- `between`;
- `sameAs`;
- `notSameAs`.

`EvaluateGlobal(predicates, snapshot)` проверяет условия уровня раунда. Такие условия читают только `RoundFacts` и `AntagSummary`.

`EvaluateCandidate(predicates, candidate, snapshot, selectedSlots)` проверяет условия кандидата. Такие условия читают `CandidateFacts`, `RoundFacts` и уже выбранные слоты, если используется `compareTo`.

Важный принцип: ложное условие не является исключением. Если условие валидно, но не совпало, engine возвращает reject reason. Если runtime-форма условия невалидна, engine возвращает error result, а не падает. Это позволяет wave orchestrator аккуратно записать причину отказа сценария или слота.

Для map-полей `antagSummary.byRole` и `antagSummary.byObjectiveType` отсутствующий ключ считается нулем. Это нужно, чтобы условие вроде "нет такой роли" не ломалось, если ключ просто отсутствует в словаре.

## 8. Волны распределения

За волны отвечает `IntentionsWaveOrchestrator` в `Content.Server/Intentions/Waves/IntentionsWaveOrchestrator.cs`.

Есть два вида волн:

- `Start`: стартовая выдача намерений;
- `Refill`: добор дефицита после того, как часть сценариев заморожена или отменена.

Есть два режима выполнения:

- dry-run: волна собирает результаты, но ничего не записывает в registry;
- committed run: успешные сборки сразу передаются в `IntentionsCommitService`.

Публичные методы orchestrator:

- `RunStartWave`;
- `RunStartWaveAndCommit`;
- `RunRefillWave`;
- `RunRefillWaveAndCommit`.

Автоматический стартовый запуск находится в `IntentionsDistributionSystem`. Он слушает `PlayerSpawnCompleteEvent`, `GameRunLevelChangedEvent` и имеет дополнительный резервный запуск в `Update`, если раунд уже перешел в `InRound`, а стартовая волна еще не была даже запланирована. Это сделано, чтобы стартовая волна не терялась из-за порядка событий на старте раунда.

После планирования система ждет небольшую задержку, строит snapshot и запускает `RunStartWaveAndCommit`. Если snapshot пустой или временно не собрался, система делает несколько попыток. Это нужно для случаев, когда игрок уже появился, но mind, профиль или тело еще не полностью готовы для snapshot.

После успешной стартовой committed-волны `IntentionsDistributionSystem` планирует следующую refill-волну через случайный интервал от 8 до 15 минут. Перед refill-волной тот же системный владелец вызывает `ReconcileBeforeRefillNow()`, затем строит refill-snapshot и только после этого запускает committed refill.

И автоматический, и безопасный ручной путь оставляют серверные сообщения до и после старта/завершения start/refill-волн. После успешного commit система обновляет открытые окна намерений для затронутых mind, проигрывает `/Audio/Machines/quickbeep.ogg` каждому персонажу, который получил новые намерения, один раз на mind, и отправляет тому же игроку одно приватное server-chat сообщение со сводкой выдачи.

Чат-уведомление строится не по шаблонам напрямую, а через отдельный hidden-safe helper. Visible intentions перечисляются по локализованным `nameLoc`, а hidden intentions до reveal не раскрываются по имени и учитываются только как count. Если одному `mind` в одном commit batch выдали несколько карточек, в чат уходит одно агрегированное сообщение.

Отдельно от обычных волн теперь существует debug forced-assign path. Он не принадлежит `IntentionsDistributionSystem`, не двигает `_nextWaveId`, не трогает автотаймеры и пишет свои wave contexts под отрицательными manual `waveId`.

### Квоты

Квоты считает `IntentionsQuotaCalculator`.

Для `fixed`:

```text
targetQuota = value
```

Для `ratio`:

```text
targetQuota = Floor(distributionCrewBaseline * ratio)
```

Для `clamp`:

```text
baseQuota = Floor(distributionCrewBaseline * ratio)
targetQuota = Clamp(baseQuota, min, max)
```

В текущей реализации `min` и `max` относятся именно к `clamp`. У обычного `ratio` они не используются.

Для start-волны `distributionCrewBaseline` равен текущему `CrewCount` из snapshot.

Для refill-волны baseline равен максимуму из предыдущих `DistributionCrewBaseline` и текущего `CrewCount`. Это защищает refill от ситуации, когда экипаж уменьшился, и система начинает считать, что раньше выданные сценарии больше не нужны.

### Порядок категорий

Категории сортируются так:

1. Больший `priority` раньше.
2. При равном `priority` используется порядок объявления.

Внутри категории сценарии хранятся в declaration order, но выбираются через deterministic weighted random по `weight`.

### Seed и детерминизм

Если request содержит seed, используется он. Если seed не задан, он строится стабильно из:

- `waveId`;
- `snapshotId`;
- `gameMode`;
- `stationTime.Ticks`.

Нельзя использовать обычный `string.GetHashCode`, потому что он не предназначен для стабильного воспроизведения между запусками. Для этого есть `IntentionsDeterministicRandom.BuildSeed`.

### Pool сценариев

В pool попадают только сценарии, которые:

- валидны;
- включены через `enabled`;
- принадлежат текущей категории;
- еще не назначались в этом раунде;
- проходят `globalPredicates`.

Если global predicates не прошли, сценарий получает reject reason и не идет в builder.

### Scenario builder

`IntentionsScenarioBuilder` собирает один сценарий по `SlotBuildOrder`, который пришел из validation.

Правила builder:

- owner собирается первым;
- `bindToSlot` копирует уже выбранного кандидата из другого слота;
- `allowSameActorAs` разрешает вернуть уже выбранного кандидата в pool для этого слота;
- обычные слоты не могут повторно использовать reserved mind ids;
- optional slot без кандидата получает `Skipped`, но сценарий остается успешным;
- required slot без кандидата делает весь сценарий неуспешным;
- `compareTo` проверяется через predicate engine по уже выбранному слоту;
- primary slot учитывает `effectiveMaxPrimaryPerMind`;
- builder не меняет registry, квоты, индексы, mind links и UI.

Для debug forced assign используется отдельный явный builder path в `IntentionsRuntimeAdminService`. Он повторяет те же slot и predicate проверки, но берет назначение mind ids из аргументов команды, а не выбирает кандидатов сам.

### Refill

Для refill target считается не как полная квота, а как дефицит:

```text
refill target = max(0, full target quota - existing Active/Frozen count)
```

`Cancelled` не считается заполнением квоты. При этом `AssignedScenarioIds` не очищается после cancellation, поэтому один `scenarioTemplate` не назначается повторно в этом же раунде.

## 9. Runtime Registry и Commit

`IntentionsRuntimeRegistry` - это in-memory хранилище текущего раунда. Оно находится под серверным контролем и принадлежит `IntentionsRuntimeSystem`.

Основные индексы registry:

- `ScenarioByUid`: scenario uid -> scenario instance;
- `IntentionByUid`: intention uid -> intention instance;
- `IntentionIdsByMind`: mind id -> intention ids;
- `ScenarioUidByIntentionUid`: intention uid -> scenario uid;
- `SlotAssignmentByScenarioAndSlot`: scenario uid + slot id -> slot assignment;
- `AssignedScenarioIds`: scenarioTemplate ids, уже использованные в раунде;
- `AssignedPrimaryByMind`: счетчик primary ownership по mind и category;
- `WaveContextByWaveId`: сохраненные контексты волн;
- `HiddenIntentionsByRevealTime`: hidden timer index;
- `MissingOwnerScenarioIds`: сценарии с owner slot в `Missing`.

`IntentionsRuntimeSystem` сейчас умеет не только выдавать registry, но и:

- `ResetRegistry()` — заменить runtime registry целиком и вернуть список затронутых mind;
- `NextManualWaveId()` — выдать следующий отрицательный debug `waveId` для принудительного назначения.

UID runtime-объектов round-scoped и монотонные:

- `ScenarioInstanceUid`;
- `IntentionInstanceUid`.

Они не являются глобальными идентификаторами и не сохраняются между раундами.

Текущие runtime-статусы:

- `ScenarioRuntimeStatus`: `Active`, `Frozen`, `Cancelled`;
- `IntentionRuntimeStatus`: `Active`, `Cancelled`;
- `ScenarioSlotAssignmentStatus`: `Assigned`, `Missing`, `Invalidated`.

`IntentionsCommitService` отвечает за атомарную запись successful `ScenarioBuildResult` в registry.

Commit flow:

1. Проверить входы без изменения registry.
2. Убедиться, что build успешный.
3. Убедиться, что scenario id и category совпадают с template.
4. Проверить, что scenarioTemplate еще не назначался.
5. Проверить owner slot.
6. Проверить, что все built slots существуют в template.
7. Найти все intention templates в `ValidationCatalog`.
8. Разрешить text bindings.
9. Разрешить `copyableTextResolved`.
10. Подготовить `ScenarioInstance`, `ScenarioSlotAssignment`, `IntentionInstance` локально.
11. Начать запись в registry.
12. После каждой записи добавить rollback action.
13. При ошибке выполнить rollback в обратном порядке.

Rollback не обязан откатывать счетчики UID, но обязан удалить все runtime-объекты и индексы, которые успел добавить commit.

`ScenarioInstance` хранит:

- uid;
- id исходного scenario template;
- category id;
- status;
- owner slot id;
- owner mind id;
- owner entity uid;
- wave id;
- время создания;
- slot assignments.

`ScenarioSlotAssignment` хранит:

- scenario uid;
- slot id;
- kind;
- mind id;
- owner entity uid;
- status;
- intention uid;
- required;
- информацию о bind;
- время назначения;
- время последнего изменения статуса;
- причину изменения.

`IntentionInstance` хранит:

- uid;
- intention template id;
- scenario uid;
- slot id;
- owner mind id;
- owner entity uid;
- kind;
- status;
- время назначения;
- время изменения статуса;
- hidden/visible state;
- reveal mode;
- reveal time;
- `ResolvedTextParameters`;
- `CopyableTextResolved`.

### Text bindings

Text bindings вычисляются при commit один раз. UI потом читает готовые параметры и не пересчитывает их из snapshot.

Поддерживаемые источники:

- `self`: поле текущего кандидата;
- `slot`: поле кандидата из другого built slot;
- `round`: `stationName` или `stationTime`;
- `literal`: значение из YAML.

Если binding ссылается на неизвестное поле, commit завершается ошибкой до изменения registry.

### Visibility и reveal index

Base visibility берется из `IntentionTemplate.DefaultVisibility`. `ScenarioEntry.VisibilityOverride` может заменить ее.

Visible intention не получает reveal timer. Hidden intention без reveal или с `none` остается hidden без `RevealedAtRoundTime`. Hidden intention с `timer` получает `RevealedAtRoundTime = waveContext.StartedAtRoundTime + minutes` и попадает в `HiddenIntentionsByRevealTime`.

Поверх обычного commit-path теперь есть отдельный runtime admin mutation слой:

- `IntentionsRuntimeAdminService.ClearAllScenarios(...)` физически сбрасывает весь runtime и все индексы;
- `IntentionsRuntimeAdminService.RemoveScenario(...)` физически purge-ит один сценарий и связанные intentions/indexes;
- `IntentionsRuntimeAdminService.TryForceAssignScenario(...)` строит и коммитит один сценарий по явной slot assignment map;
- `IntentionsRuntimeAdminService.RevealHiddenIntention(...)` принудительно раскрывает одно hidden-намерение по `intentionUid` и `ownerMindId`;
- `IntentionsRuntimeAdminService.RevealAllHiddenIntentionsForMind(...)` принудительно раскрывает все active hidden-намерения одного `ownerMindId`.

Для `remove` важны две детали:

- `AssignedScenarioIds` снимается только если в runtime больше не осталось другого сценария с тем же template id;
- `AssignedPrimaryByMind` декрементится только если удаляемый сценарий еще удерживал live-квоту (`Active` или `Frozen`).

Для admin reveal важны три детали:

- команда работает только с уже созданными runtime `IntentionInstance`, а не с шаблонами;
- раскрываются оба текущих hidden-варианта MVP: `reveal = none` и `reveal = timer`;
- операция меняет только hidden-state и membership в `HiddenIntentionsByRevealTime`, не трогая lifecycle сценария, квоты или wave scheduler.

## 10. Lifecycle и Refill

Lifecycle отвечает за ситуацию, когда владелец primary-намерения пропал, вернулся или окончательно недоступен.

Основные файлы:

- `Content.Server/Intentions/Runtime/IntentionsLifecycleModels.cs`;
- `Content.Server/Intentions/Runtime/IntentionsLifecycleService.cs`;
- `Content.Server/Intentions/Runtime/IntentionsLifecycleSystem.cs`.

`OwnerAvailabilityResult` описывает доступность владельца:

- `Available`;
- `TemporarilyMissing`;
- `PermanentlyUnavailable`.

Технически `IntentionsLifecycleSystem` считает владельца доступным, если mind существует, у него есть `UserId`, сессия находится в игре, `TimeOfDeath == null`, и есть живой `OwnedEntity`.

Временно недоступен:

- сессия не in-game;
- нет текущего тела;
- тело удаляется;
- mind временно без тела.

Окончательно недоступен:

- mind не найден;
- нет `MindComponent`;
- нет `UserId`;
- `TimeOfDeath != null`.

### Текущая двухволновая схема

Важный текущий инвариант: 30-секундная проверка не переводит scenario в `Frozen` и не переводит его в `Cancelled`.

30-секундная проверка делает только:

```text
owner slot -> Missing
```

Это выполняется через `MarkOwnerMissing`. Сценарий при этом остается в текущем состоянии, если он не был `Cancelled`. Такой подход нужен, чтобы не раскрывать метаинформацию слишком быстро. Если игрок исчез или умер прямо перед refill-волной, система сначала помечает только слот, а не отменяет весь сценарий сразу.

Ключевое ограничение текущей реализации: lifecycle привязан именно к owner slot и `ScenarioInstance.OwnerMindId`. Уход partner/observer или другого secondary-участника не переводит сценарий в `Frozen` и не отменяет его автоматически.

Перед committed refill-волной команда `intentions.wave.run refill ...` вызывает `IntentionsLifecycleSystem.ReconcileBeforeRefillNow()`. Этот метод запускает `IntentionsLifecycleService.ReconcileBeforeRefill(...)` до построения нового snapshot.

Внутри refill pre-pass есть два шага.

Шаг 1: сценарии, которые уже были `Frozen` на момент запуска текущей refill-волны.

- Если владелец снова доступен: `Frozen -> Active`, owner slot возвращается в `Assigned`.
- Если владелец все еще недоступен: `Frozen -> Cancelled`, все `ScenarioSlotAssignment` переводятся в `Invalidated`, все связанные `IntentionInstance` переводятся в `Cancelled`.

Шаг 2: сценарии, которые не находятся в `Frozen`, но owner slot уже находится в `Missing`.

- Если владелец снова доступен: owner slot возвращается в `Assigned`, scenario остается или становится `Active`.
- Если владелец все еще недоступен: `Active + Missing -> Frozen`.

Таким образом, обычная цепочка выглядит так:

```text
30-секундная проверка: owner slot -> Missing
первая refill-волна: Active + Missing -> Frozen
следующая refill-волна: Frozen -> Cancelled, если владелец не вернулся
```

Если владелец вернулся до refill, система может восстановить owner slot в `Assigned` и обновить `OwnerEntityUid`.

### Освобождение квоты

Когда сценарий становится `Cancelled`:

- scenario status становится `Cancelled`;
- все slot assignments становятся `Invalidated`;
- все intention instances становятся `Cancelled`;
- hidden reveal entries удаляются;
- `AssignedPrimaryByMind` уменьшается;
- `MissingOwnerScenarioIds` очищается для этого scenario uid;
- `AssignedScenarioIds` не очищается.

Это означает, что квота категории освобождается, но тот же `scenarioTemplate` не будет назначен повторно в текущем раунде.

### Защита от мета-знания

Если бы 30-секундная проверка сразу делала `Cancelled`, secondary-участник мог бы увидеть отмену своего намерения и понять, что владелец, вероятно, умер или исчез. Поэтому текущая схема откладывает переходы scenario-level до refill. Сначала меняется только owner slot, затем на следующей волне scenario замораживается, и только еще через одну refill-волну отменяется, если владелец не вернулся.

## 11. Reveal

Reveal отвечает за раскрытие hidden-намерений. В текущем MVP есть два рабочих режима:

- `none`;
- `timer`.

Основные файлы:

- `Content.Server/Intentions/Runtime/IntentionsRevealService.cs`;
- `Content.Server/Intentions/Runtime/IntentionsRevealSystem.cs`.

Hidden-намерение с timer при commit попадает в `HiddenIntentionsByRevealTime`. `IntentionsRevealSystem` периодически смотрит текущее время раунда и вызывает `IntentionsRevealService.EvaluateTimerReveals(...)`.

Reveal service:

- берет все hidden intentions, у которых наступило время reveal;
- пропускает отмененные намерения;
- меняет hidden state на revealed;
- удаляет intention из reveal index;
- возвращает список раскрытых intentions.

После раскрытия reveal system обновляет UI для затронутых mind. UI не раскрывает hidden-данные самостоятельно. До reveal игрок получает только redacted card data: безопасный заголовок, безопасное описание и текст о reveal.

Клиент может локально тиканьем обновлять уже разрешенный reveal timer в detail/list UI, но authoritative переход hidden -> visible по-прежнему происходит только на сервере через reveal service.

Отдельно от автоматического timer reveal теперь существует admin debug-path. Он умеет:

- раскрыть одно hidden intention по `intentionUid`;
- раскрыть все active hidden intentions выбранного `ownerMindId`;
- удалить timer-hidden intentions из reveal index при досрочном раскрытии;
- сразу обновить открытый UI через `IntentionsUiSystem.RefreshMind(...)`;
- показать владельцу popup через обычный loc key `intentions-ui-reveal-notification`.

Если команда раскрывает несколько hidden intentions одного mind, popup отправляется только один раз на mind. Этот путь не является новой игровой механикой reveal: это отдельная admin/debug mutation поверх уже существующего runtime.

## 12. UI и Read Model

UI устроен как read-only представление runtime-состояния.

Основные файлы:

- `Content.Server/Intentions/UI/IntentionsQueryService.cs`;
- `Content.Server/Intentions/UI/IntentionsUiSystem.cs`;
- `Content.Server/Intentions/UI/IntentionsEui.cs`;
- `Content.Server/Intentions/UI/IntentionsAdminVerbSystem.cs`;
- `Content.Client/Intentions/UI/IntentionsEui.cs`;
- `Content.Client/Intentions/UI/IntentionsUIController.cs`;
- `Content.Client/Intentions/UI/IntentionsListCardControl.xaml(.cs)`;
- `Content.Client/Intentions/UI/IntentionsTextHighlighting.cs`;
- `Content.Shared/Intentions/UI/IntentionsUiModels.cs`.

`IntentionsQueryService` строит `IntentionsEuiState`. Он берет `ValidationCatalog`, `IntentionsRuntimeRegistry`, target mind и режим UI.

Есть два режима:

- player view;
- admin read-only view.

Player view открывается самим игроком кнопкой в верхнем меню. Admin read-only view открывается отдельно: `IntentionsAdminVerbSystem` добавляет администратору verb в ПКМ-меню по персонажу-цели, если у цели есть `mind`, а у вызывающего пользователя есть права администратора. Действие verb вызывает `IntentionsUiSystem.OpenAdminUi(...)` и открывает окно намерений именно для target `mind`, а не для локального игрока администратора.

Для player view hidden-намерения редактируются до отправки клиенту:

- template id скрывается;
- scenario id скрывается;
- category id скрывается;
- реальное название не отправляется;
- реальное описание не отправляется;
- copyable text не отправляется;
- icon/color заменяются безопасными значениями;
- `ResolvedTextParameters` очищаются, чтобы клиент не мог подсветить или восстановить скрытые подстановки.

Для admin view карточки показываются подробнее, но UI остается read-only: админский просмотр не меняет runtime state. Через этот режим администратор может посмотреть намерения выбранного персонажа напрямую из ПКМ-меню, не переключая обычное player-окно и не подменяя локальный player context.

`IntentionsCardView` хранит уже локализованные и безопасные для показа данные карточки. Кроме обычных текстов она несет `Author`, `Status`, `Icon`, `Color`, `CopyableText` и `ResolvedTextParameters`. Эти параметры нужны клиенту только для визуального выделения уже подставленных частей текста. Клиент не вычисляет их заново и не локализует карточку повторно.

`IntentionsUiSystem` открывает UI для игрока или администратора, строит состояние и умеет обновлять mind после commit или reveal. `IntentionsEui` на сервере обрабатывает открытие материалов и пересылку актуального состояния в уже открытые окна.

Клиентский `IntentionsUIController` подключает окно к отдельной кнопке в верхнем меню игры. Сам `IntentionsEui` показывает два прокручиваемых списка карточек: основные и второстепенные.

Текущий порядок карточек в player UI задается на сервере в `IntentionsQueryService` и клиентом не пересчитывается:

1. `ScenarioStatus`: `Active`, затем `Cancelled`, остальные статусы после них;
2. `HiddenRank`: visible раньше hidden;
3. для visible-карточек — `CategoryPriority` по убыванию;
4. затем `CategorySortName` по алфавиту;
5. затем стабильные tie-breakers по времени назначения и `IntentionUid`.

Для hidden player-карточек category-сортировка намеренно не используется: они уводятся в хвост своего status-блока, чтобы не выдавать категориальную метаинформацию через порядок.

Каждая строка списка строится через `IntentionsListCardControl`: слева цветная полоса категории, затем иконка, название и вторая строка текста. Для hidden player-карточек вместо реальных данных показываются безопасная подпись и фиксированная иконка замка, а вторая строка используется как reveal line. Если у карточки есть `RevealedAtRoundTime`, клиент локально тикает таймер до следующего серверного refresh.

`SetSelected()` в `IntentionsListCardControl` теперь делает programmatic press-state update тихо через `MuteSounds`, чтобы 30-секундные или reveal/passive refresh не воспроизводили click sound повторно. Реальный пользовательский клик звук сохраняет.

Правая часть окна сейчас собрана как рамочное detail-view:

- header с заголовком слева и status badge справа;
- строка `Created by ...`;
- optional reveal block;
- блок `Intention`;
- блок `OOC note`;
- нижняя зона с `Open materials`.

Фоновая заливка используется только у status badge и `OOC note`. `Reveal` идет перед описанием намерения. Краткое описание справа отдельно не повторяется.

Окно материалов — это отдельное клиентское окно с одним read-only редактором текста без бумажной рамки. Его можно растягивать, текст внутри можно выделять и копировать, но изменения не сохраняются. Повторное открытие не должно плодить новые экземпляры этого окна. У контейнера окна есть небольшой правый inset, чтобы вертикальный скроллбар не закрывал текст.

`IntentionsTextHighlighting` используется поверх уже готовых строк. Он находит значения из `ResolvedTextParameters` и выделяет их более жирным начертанием в видимом тексте карточки. На hidden player-карточках этот механизм ничего не подсвечивает, потому что набор параметров туда не передается.

### 13.4 `intentions.wave.preview <start|refill> [waveId] [seed]`

**Команда**
`intentions.wave.preview <start|refill> [waveId] [seed]`

**Назначение**
Запускает dry-run одной start/refill-волны без commit и без изменения runtime registry.

**Входные параметры**
- `start|refill` — обязательный параметр, тип: строка, допустимые значения: `start` или `refill`.
- `waveId` — необязательный параметр, тип: `int`; по умолчанию `1`.
- `seed` — необязательный параметр, тип: `int`; если не задан, используется auto seed.

**Выход**
Печатает `wave preview`:
- общую шапку волны (`waveId`, kind, status, seed, snapshot, startedAt);
- round crew/baseline;
- per-category state: `target`, `filled`, `desired`, `activeFrozen`, `refill`, `maxPrimaryPerMind`, `exhausted`, `reason`;
- pool/rejected scenario ids;
- successful builds;
- scenario rejects и slot rejects.

| Поле | Описание |
|---|---|
| `target` | Квота, которую текущая волна пытается заполнить для этой категории. Для `start` это обычная целевая квота категории. Для `refill` это уже не полная квота, а дефицит после учета существующих `Active + Frozen`. |
| `filled` | Сколько успешных сценариев этой категории может быть собрано внутри текущей preview-волны. |
| `desired` | Полная желаемая квота категории до применения refill-дефицита. Для `start` обычно совпадает с `target`. Для `refill` показывает исходную полную цель категории, даже если реально добирается только часть. |
| `activeFrozen` | Сколько runtime-сценариев этой категории уже существует в состояниях `Active` или `Frozen` на момент refill-расчета. Для `refill` именно это число считается уже занятым местом в квоте. `Cancelled` сюда не входит. |
| `refill` | Сколько новых сценариев refill-волна пытается добавить для этой категории. По смыслу это `refill-target`, то есть недостающее количество после учета `Active + Frozen`. |
| `maxPrimaryPerMind` | Эффективный лимит, сколько `primary`-сценариев этой категории один `mind` может иметь как владелец. |
| `exhausted` | Исчерпана ли категория в рамках текущей волны. `yes` означает, что wave orchestrator больше не будет пытаться собрать для нее новые сценарии в этой волне. `no` означает, что категория еще может участвовать в выборе. |
| `reason` | Причина, по которой категория стала исчерпанной (`exhausted`). Возможные значения: `None` — категория еще не исчерпана; `Blocked` — категория заблокирована и не может участвовать в выборе; `PoolEmpty` — в пуле не осталось подходящих сценариев; `PoolExhausted` — пул сценариев был исчерпан в процессе попыток; `QuotaFilled` — текущая wave-квота категории уже заполнена. |

**Особенности**
Для `start` dry-run использует текущие `AssignedScenarioIds` и `AssignedPrimaryByMind` из registry. Для `refill` считает дефицит по текущему registry, но ничего не коммитит.

### 13.5 `intentions.wave.run <start|refill> [seed]`

**Команда**
`intentions.wave.run <start|refill> [seed]`

**Назначение**
Безопасный committed path через `IntentionsDistributionSystem`.

**Входные параметры**
- `start|refill` — обязательный параметр, тип: строка, допустимые значения: `start` или `refill`.
- `seed` — необязательный параметр, тип: `int`; если не задан, используется auto seed.

**Выход**
Принимает scheduler-aware решение и печатает:
- строку о запрошенном safe run;
- итоговое сообщение distribution system;
- если волна реально была запущена, `wave preview` результата commit;
- `registry dump` после commit;
- текущее состояние `intentions.wave.timer`.

**Особенности**

- При невалидных аргументах или legacy syntax команда печатает `Help`/ошибку парсинга.

### 13.9 `intentions.registry.dump`

**Команда**
`intentions.registry.dump`

**Назначение**
Показывает полное текущее состояние runtime-реестра Intentions. Показывает, что именно сейчас сохранено на сервере.

**Входные параметры**
Параметров нет.

**Выход**
Команда печатает подробный `registry dump`, в который входят:

  - намерений;
  - записей в mind-индексе;

- В `player` mode вывод hidden-safe и соответствует redacted player read-model.
- В `admin` mode вывод read-only, но без player redaction.

### 13.11 `intentions.reveal.dump`

**Команда**
`intentions.reveal.dump`

**Назначение**
Печатает timer index для hidden-намерений.

**Входные параметры**
Параметров нет.

**Выход**
Печатает `reveal dump`:
- текущее round time;
- reveal buckets по `revealAt`;
- состояние bucket (`due` или `pending`);
- связанные runtime intentions с `template`, `scenarioUid`, `mind`, `hidden`, `status`, `mode`.

**Особенности**
Если timer index пуст, команда явно пишет, что `Hidden timer index: empty`.

### 13.12 `intentions.category.quota [waveId]`

**Команда**
`intentions.category.quota [waveId]`

**Назначение**
Показывает per-category quota state из уже сохранённого `DistributionWaveContext`.

**Входные параметры**
- `waveId` — необязательный параметр, тип: `int`.

**Выход**
Печатает `category quota dump`:
- шапку волны: `waveId`, kind, status, baseline, seed;
- по каждой категории: `target`, `filled`, `desired`, `activeFrozen`, `refill`, `maxPrimaryPerMind`, `exhausted`, `reason`.

**Особенности**
- Команда не пересчитывает квоты заново, а читает уже сохранённый `WaveContextByWaveId`.
- Если `waveId` не передан, берётся контекст с **наибольшим `waveId`** из `WaveContextByWaveId`.
- Если контекст не найден, печатается `No wave context is available.`

### 13.13 `intentions.logs.trace <waveId>`

**Команда**
`intentions.logs.trace <waveId>`

**Назначение**
Печатает lightweight trace по одной сохранённой волне.

**Входные параметры**
- `waveId` — обязательный параметр, тип: `int`.

**Выход**
Печатает `wave trace`:
- шапку волны: `waveId`, kind, status, snapshot, seed, startedAt;
- crew facts (`active`, `baseline`);
- timeline с successful builds и reject reasons;
- committed scenarios этого `waveId`, если они есть в registry.

**Особенности**
Если `waveId` не найден в `WaveContextByWaveId`, команда печатает отдельное сообщение об отсутствии контекста.

### 13.14 `intentions.runtime.clearall`

**Команда**
`intentions.runtime.clearall`

**Назначение**
Полностью сбрасывает Intentions runtime registry и все связанные индексы.

**Входные параметры**
Параметров нет.

**Выход**
Печатает итоговую строку:
- сколько runtime scenarios удалено;
- сколько runtime intentions удалено.

**Особенности**
- Команда физически заменяет registry на новый пустой экземпляр.
- Она очищает `ScenarioByUid`, `IntentionByUid`, `IntentionIdsByMind`, `ScenarioUidByIntentionUid`, `SlotAssignmentByScenarioAndSlot`, `AssignedScenarioIds`, `AssignedPrimaryByMind`, `WaveContextByWaveId`, `HiddenIntentionsByRevealTime`, `MissingOwnerScenarioIds`.
- Scheduler state в `IntentionsDistributionSystem` не трогается.
- После очистки refresh-ит UI затронутых mind.

### 13.15 `intentions.runtime.remove <scenarioUid> <ownerMindId>`

**Команда**
`intentions.runtime.remove <scenarioUid> <ownerMindId>`

**Назначение**
Физически удаляет один runtime scenario и все его committed intentions.

**Входные параметры**
- `scenarioUid` — обязательный параметр, тип: `ScenarioInstanceUid` в plain integer syntax.
- `ownerMindId` — обязательный параметр, тип: `EntityUid` в plain integer syntax.

**Выход**
Печатает либо:
- сообщение об успешном удалении;
- либо `WriteError` с причиной отказа.

**Особенности**
- `ownerMindId` используется как дополнительный safety guard.
- Команда обновляет UI всех затронутых mind после успешного purge.
- Это physical remove, а не перевод сценария в `Cancelled`.

### 13.16 `intentions.runtime.assign [--ignore-predicates] <scenarioTemplateId> <mind-or-dash>...`

**Команда**
`intentions.runtime.assign [--ignore-predicates] <scenarioTemplateId> <mind-or-dash>...`

**Назначение**
Принудительно выдает один конкретный сценарий вручную, без обычного автоматического выбора через волну распределения.

Эта команда нужна для отладки, ручной проверки шаблонов и точечного тестирования сценария на конкретных игроках. Она позволяет явно указать, какие `mind` должны занять нужные слоты сценария.

**Входные параметры**

- `--ignore-predicates` — необязательный флаг. Если указан, команда пропускает:
  - `globalPredicates` сценария;
  - `candidatePredicates` слотов.

  Флаг разрешен только перед `scenarioTemplateId`.

- `scenarioTemplateId` — обязательный параметр, строковый идентификатор сценария (`scenarioTemplate`), который нужно назначить.

- `mind-or-dash` — обязательный позиционный набор аргументов для слотов сценария.
  Аргументы передаются **только для `non-bound` слотов** и строго **в порядке `validated slotBuildOrder`**.

  Каждый аргумент может быть:
  - `EntityUid` в plain integer syntax — если в слот нужно явно назначить конкретного игрока;
  - `-` — если слот optional и его нужно пропустить.

**Как подбирается порядок аргументов**

Команда не принимает `slotId` по именам. Она читает уже провалидированный `slotBuildOrder` сценария и ждёт аргументы именно в этом порядке.

При этом:
- слоты с `bindToSlot` **не передаются вручную**;
- такие слоты будут заполнены автоматически на основе уже указанного другого слота;
- optional-слот можно пропустить через `-`;
- required-слот нельзя пропустить через `-`.

**Выход**

Если назначение прошло успешно, команда печатает:

- краткое итоговое сообщение об успешном назначении;
- подробный `runtime assign` debug output, в котором обычно видны:
  - `scenarioTemplateId`;
  - использованный `waveId`;
  - `ignoredPredicates=yes|no`;
  - ожидаемая структура аргументов слотов;
  - результат сборки сценария;
  - результат commit;
  - созданные `scenarioUid` и `intentionUid`.

Если назначение не удалось, команда печатает:

- `WriteError` с основной причиной ошибки;
- если доступна подробная диагностика, дополнительно печатает structured output с причинами отказа:
  - global predicate rejects;
  - slot rejects;
  - mismatch по структуре аргументов;
  - ошибки по candidate predicates;
  - ошибки по `allowSameActorAs` / `bindToSlot`.

**Особенности**

- Команда использует отдельный отрицательный `manual waveId`, чтобы не конфликтовать с обычными start/refill-волнами.
- После успешного commit игрок получает тот же приватный chat-summary о новых карточках, что и при обычной автоматической выдаче.
- Она **обходит только**:
  - quota категории;
  - `maxPrimaryPerMind`.
- Если задан `--ignore-predicates`, команда дополнительно обходит:
  - `globalPredicates`;
  - `candidatePredicates`.
- Она **не обходит**:
  - `AssignedScenarioIds`;
  - правило уникальности сценария в пределах раунда;
  - проверку присутствия кандидата в текущем snapshot;
  - `allowSameActorAs`;
  - `bindToSlot`;
  - required/optional-правила слотов.
- `bindToSlot` подставляется автоматически и не требует отдельного аргумента.
- `-` разрешён только для optional-слота.
- Без флага `--ignore-predicates` система по-прежнему проверяет и `globalPredicates`, и `candidatePredicates`.
- Даже при `--ignore-predicates` система всё равно проверяет:
  - структуру слотов;
  - `allowSameActorAs`;
  - корректность `bindToSlot`;
  - наличие кандидата в текущем snapshot.

**Пример**

Допустим, есть сценарий с таким порядком `non-bound` слотов:

1. `owner`
2. `partner`
3. `observer` (optional)

Тогда команда может выглядеть так:

```text
intentions.runtime.assign IntentionsTestSocialTwoPerson 5129 5313 -
```

Если нужно назначить тот же сценарий, но временно проигнорировать `globalPredicates` и `candidatePredicates`, вызов будет таким:

```text
intentions.runtime.assign --ignore-predicates IntentionsTestSocialTwoPerson 5129 5313 -
```

В этом режиме команда все равно не назначит сценарий повторно в том же раунде и не разрешит невалидную структуру слотов или запрещенное повторное использование одного и того же `mind`.

### 13.17 `intentions.runtime.reveal one <intentionUid> <ownerMindId>` / `intentions.runtime.reveal all <ownerMindId>`

**Команда**
`intentions.runtime.reveal one <intentionUid> <ownerMindId>`
`intentions.runtime.reveal all <ownerMindId>`

**Назначение**
Принудительно раскрывает hidden runtime-намерения без изменения lifecycle сценария.

**Входные параметры**
Для режима `one`:
- `intentionUid` — обязательный параметр, тип: `IntentionInstanceUid` в plain integer syntax;
- `ownerMindId` — обязательный параметр, тип: `EntityUid` в plain integer syntax.

Для режима `all`:
- `ownerMindId` — обязательный параметр, тип: `EntityUid` в plain integer syntax.

**Выход**
На успехе печатает:
- итоговое сообщение;
- `runtime reveal` structured output со scope, target mind/intention, count раскрытых intentions и списком `RevealedIntentions`.

На ошибке печатает:
- `WriteError` с причиной;
- structured `runtime reveal` output с failure metadata.

**Особенности**
- `one` использует `ownerMindId` как safety guard.
- `all` раскрывает только active hidden intentions указанного владельца.
- Команда работает и для `reveal = none`, и для `reveal = timer`.
- Visible и cancelled intentions не меняются.
- Для timer-hidden intentions очищается запись из `HiddenIntentionsByRevealTime`.
- После успеха refresh-ится player UI и отправляется popup; если раскрыто несколько intentions одного mind, popup идёт один раз на mind.

## 16. Основные потоки данных

### Стартовая выдача

```text
YAML/FTL
-> prototype manager
-> IntentionsValidationService
-> ValidationCatalog
-> IntentionsSnapshotService
-> IntentionsSnapshot
-> IntentionsWaveOrchestrator
-> IntentionsScenarioBuilder
-> IntentionsCommitService
-> IntentionsRuntimeRegistry
-> IntentionsQueryService
-> EUI
-> карточки игрока
```

На старте автоматический путь запускается через `IntentionsDistributionSystem`: после спавна игрока, после перехода раунда в `InRound` или через резервную проверку в `Update`. Ручной безопасный путь идет через `intentions.wave.run start`, а низкоуровневый тестовый обходной путь — через `intentions.test.wave.run start`.

### Refill

```text
runtime registry
-> ReconcileBeforeRefill
-> snapshot после lifecycle pre-pass
-> расчет Active/Frozen count
-> расчет refill deficit
-> выбор новых сценариев
-> builder
-> commit
-> UI refresh
```

Refill не изменяет существующие `Active`/`Frozen` instances, кроме lifecycle pre-pass перед snapshot.

### Скрытая карточка

```text
commit
-> hidden intention
-> HiddenIntentionsByRevealTime
-> IntentionsRevealSystem
-> IntentionsRevealService
-> intention becomes revealed
-> UI refresh
```

До reveal player UI получает redacted карточку. После reveal query service начинает отдавать настоящий текст.

### Пропавший владелец

```text
30-секундная проверка
-> owner slot Missing
-> первая refill-волна
-> Active + Missing -> Frozen
-> следующая refill-волна
-> Frozen -> Cancelled или Frozen -> Active
```

Если владелец вернулся, система обновляет owner entity uid и возвращает слот в `Assigned`.

## 17. Частые точки изменения

### Как добавить новое поле в шаблон

1. Добавить поле в нужный prototype class в `Content.Shared/Intentions/Prototypes`.
2. Добавить validation-правило в `IntentionsValidationService`.
3. Если поле влияет на runtime, добавить его в соответствующую runtime-модель.
4. Если поле нужно UI, добавить его в `IntentionsUiModels` и `IntentionsQueryService`.
5. Добавить YAML/FTL пример.
6. Добавить тесты validation и runtime-поведения.

### Как добавить новый predicate field

1. Добавить поле в `IntentionsPredicateSchema`.
2. Добавить чтение поля в `IntentionsPredicateEngine`.
3. Если поле приходит из живой игры, добавить его в `CandidateFacts` или `RoundFacts`.
4. Добавить сбор поля в `IntentionsSnapshotService`.
5. Добавить нормализацию в `IntentionsSnapshotFactory`.
6. Добавить validation tests и predicate tests.

### Как добавить новый predicate operator

1. Добавить оператор в `IntentionsPredicateSchema`.
2. Добавить обработку в `IntentionsPredicateEngine`.
3. Добавить validation-ограничения для value/values/range/compareTo.
4. Добавить тесты на успешный match, false-result и invalid runtime form.

### Как поменять квоты

Квоты находятся в `ScenarioCategoryPrototype` и считаются через `IntentionsQuotaCalculator`. Если меняется только контент, достаточно YAML. Если меняется формула, надо менять calculator, wave tests и refill tests.

### Как добавить новое состояние lifecycle

1. Добавить статус в shared runtime models.
2. Добавить copy/replace-методы, если состояние меняет runtime instance.
3. Обновить `IntentionsLifecycleService`.
4. Обновить `IntentionsRuntimeRegistry`, если нужен новый индекс.
5. Обновить query service и UI, если статус должен отображаться.
6. Добавить lifecycle/refill tests.

### Как добавить новую debug-команду

1. Добавить класс `IConsoleCommand` в `Content.Server/Intentions/Debug/IntentionsDebugCommands.cs` или отдельный файл в этой папке.
2. Пометить команду нужным admin flag.
3. Использовать существующие helpers и formatters.
4. Если вывод сложный, добавить метод в `IntentionsDebugFormatters`.
5. Добавить tests на форматирование или поведение.

### Как добавить новое поле в UI

1. Добавить поле в shared UI model.
2. Заполнить его в `IntentionsQueryService`.
3. Убедиться, что player mode не получает hidden-sensitive данные.
4. Отобразить поле в client EUI.
5. Обновить query tests и, если нужно, client UI checks.

## 18. Инварианты, которые нельзя ломать

- Builder должен оставаться без side effects.
- Commit должен быть атомарным.
- Rollback должен удалять уже записанные runtime objects и indexes.
- Один `scenarioTemplate` не должен повторяться за раунд.
- `AssignedScenarioIds` не очищается при `Cancelled`.
- `AssignedScenarioIds` можно очищать только при физическом admin purge/reset runtime.
- Hidden UI не должен читать или получать реальные скрытые тексты.
- `resolvedTextParameters` и `copyableTextResolved` вычисляются при commit, а не при открытии UI. Клиент может только подсветить уже полученные значения, но не пересчитывать текст.
- Refill считает заполнение по `Active + Frozen`.
- `Cancelled` освобождает квоту, но не разрешает повторить тот же template.
- 30-секундная проверка не переводит scenario в `Frozen`.
- 30-секундная проверка не переводит scenario в `Cancelled`.
- `Active + Missing -> Frozen` происходит только в refill pre-pass.
- `Frozen -> Cancelled` происходит только в refill pre-pass, если владелец не вернулся.
- Lifecycle сейчас owner-driven: уход partner/observer не должен случайно влиять на status scenario.
- Snapshot одной волны должен быть immutable.
- Predicate engine не должен читать живой ECS.
- Client UI не должен быть источником истины.
- Preview-команды не должны менять runtime state.
- Forced assign может обходить только category quota и `maxPrimaryPerMind`, но не round uniqueness.
- Hidden player-карточки должны оставаться в хвосте своего status-блока, чтобы порядок не раскрывал категорию.
- Programmatic UI refresh не должен производить повторный click sound.
